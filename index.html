生活小记

Posted on Dec 1 2018   |   In 生活   |  

Mina 常抱怨我毕业后不写文章了。其实当我发现不能直视当年的作品时，已经失去了自由洒脱的人格。
锅碗瓢盆的生活充满烟火气息，这时候就要少做感想，踏实奋斗，因为日子越来越复杂了。人生的前二十年都是在获得，慢慢地要学会奉献了，平衡社会关系。

有空还是要多读几本书，丰富一下自己，免得走到哪儿都是一副口若悬河的皮囊。日子嘛，可能得调整调整了，一起做些无关奋斗的有趣事情，这样闲暇的时光可以用来海阔天空地扯淡。

三十多岁后，彷佛一眼看见了远方。从儿时的小村庄，一步步走出来，看到的世界越大，心也越大。当生活一天天好起来，时代推着我们走向另一种世界，有太多东西是需要学习去面对的。人们之间变得复杂又直接，陌生又熟悉。跟少年的世界有撕裂感，变成我从来没有想象过的一种人。

我希望往后的日子，给家人更多一些牵挂，给家人更多一些关心。曾经一年吃一次肉的日子也不曾抱怨，我们现在也一定会健康快乐的。我希望以后也给 Mina 多一些支持，帮助她过的更快乐，更有意义，成就几段不可思议的人生旅程。我也希望孩子们轻松快乐地成长，寻找人生的乐趣，成为自己最想成为的人。我也希望自己能把所有的困难和不悦处理的井井有条。

愿言配德兮，携手相将。

这次交流的主题主要围绕ETL项目开发，构建，发布过程中的一些问题进行探讨，提出一些建议。

项目组织

最初的ETL项目都是独立的maven工程，导致依赖混乱。本地环境可能要从不同的目录路径下导入各种依赖模块分别编译打包。

比如ETL的依赖就包括位于不同目录下的common工程，meta工程，fw工程等。

第一次重构后形成了如下的项目结构：

  • etl
    • etl-app
    • etl-core
    • etl-controller
    • etl-execEngine
  • common
    • common-base
    • common-mt
    • common-schedule
    • common-yeast
    • …
  • meta
    • metadata
    • metadir
  • fw
    • fw-web
    • fw-admin
    • …
  • plugin
    • etl-coldb
    • etl-index
    • …

如此一来项目的内聚性提高了，项目结构也很清晰，基础组件都提炼出来了。
但是对于一个拥有众多项目的公司来说还不够，如果有个新项目的需求可能就会有人写一个新的工程，抛弃这些内部的公共组件，违反一些团队内部约定。

java项目可以通过基于maven的树状继承体系以及maven自定义初始化项目骨架来规范。

遵循development, test, product的环境区分，通过构建时添加 -P test/development来制定对于的profile以区分环境相关配置。

CI/CD

整个软件交付的周期中，通常会涉及开发人员、测试人员、运维人员分别负责研发工作、测试工作以及上线部署工作。主要问题有

  1. 代码构建的一致性和稳定性很难保证
  2. 测试环境的版本管理复杂，易出错，难回滚
  3. 生产部署环境多，配置麻烦

Continuous Integration(持续集成) Continuous Deploy/Delivery(持续部署/交付)是伴随着敏捷开发而活跃起来的概念。核心是通过一些自动化的工具
来提高构建效率，简化测试环境管理，同时使版本升级、回滚更便捷。

针对项目的情况，为了实现持续集成，目前所做的工作有：

1. 使用Jenkins持续构建项目。
2. 通过maven和spring的profile功能做环境区分（由测试人员维护测试环境的配置，实施人员维护生产环节的配置文件），在Jenkins上可以直接获取对应
环境的最终部署文件。
3. 由于目前没有使用svn， 所以的包都是从仓库分支带上不同的profile配置构建出来了，未来能用分支区分将更容易对发布版本维护。
4. 通过发布包内置tomcat实现生产环境的一键部署。

代码静态检查

这里的代码静态检查包含了代码规范检查和源代码中潜在的问题。写代码的过程是一个完全主观驱动进行的存在，不可避免地会出现一些失误或者bug，代码静态检查
可以帮助将由于疏忽造成的bug数量减少，同时保持团队统一的代码风格以提高项目的可维护性。

代码规范检查工具常用有checkstyle，在maven项目中使用可以通过插件的方式。

configLocation指定了checkstyle配置文件的路径，suppressionsLocation指定例外情况配置文件的路径。具体的配置可以参考官方文档。

findbugs是另外一个常用的代码静态检查工具，同样可以通过maven插件配置使用。

对于在IntelliJ IDEA下开发的同学，更是有findbugs的插件可以帮助提早发现代码中潜在的问题。

其他的常用工具比如PMD，SonarQube等，各位可以自行搜索了解。代码的静态检查可以作为CI/CD中的一个必须步骤，通过检查失败禁止代码提交来强制开发人员规范自己的代码。

代码审查

代码审查是指Code Review，通过让其他同事或者高级人员来复查自己的代码来控制代码质量。从短期来看可能会多占用一下开发资源，但从长远看它对团队代码质量的把控，
可维护性的提高以及团队成员综合素质和生产力的提高是有帮助的。

具体可以参考InfoQ上的文章Code review when and how

类似的文章还有很多，高质量的会继续添加。同样，代码审查可以作为开发流程的一个必须步骤，通过Jira等管理工具加入流程，在提交给测试人员测试之前强制执行。

TDD

测试驱动开发的概念已经翻来覆去很多年了，真正的实践在大团队尤其是非敏捷的团队里还是很少见的。

对于我们，可以学TDD的思想来写好测试，对于开发来说，比较重要的包括单元测试，集成测试，以及E2E测试。

好的测试覆盖可以帮助我们更自信的去重构，还可以降低回归测试的压力和成本。对于集成系统也可以降低模块之间联调的难度和成本。

本文将从源码角度解析codis的主从切换服务codis-ha。读者需要对codis的架构有所了解，并且熟悉基本的golang语法。
另外，本文假定读者熟悉golang一些常用包，比如flag，net/http，error等。

####获取源码
codis-ha的github地址为https://github.com/ngaut/codis-ha，执行

可以将codis-ha下载到本地$GOPATH/src/github.com/ngaut/codis-ha目录下。

结构为：

核心源码共五个go文件。其中main.go为入口函数(我用函数来概称一个方法/go文件/函数/程序片段)，另外四个：

  • aliverchecker.go
  • redischecker.go
  • rest.go
  • servergroup.go

除此之外，redischecker_test.go和servergroup_test.go分别为测试文件。
README.md为markdown的说明文件(这是一句废话，而且显得我很二)。

解析的顺序为从局部到整体，称之为倒叙。

####倒叙之aliverchecker.go
倒叙之aliverchecker.go只定义了一个接口AliveChecker，用来抽象实现了CheckAlive() error方法的类型

####倒叙之redischecker.go
redischecker.go定义了一个redisChecker的类型，它实现了ping() error和CheckAlive() error两个方法，所以它也一个AliveChecker类型。

####倒叙之rest.go

rest.go封装了一个发送http请求到codis-config(dashboard)的方法

####倒叙之servergroup.go

codis-ha的核心代码都在这个文件中

GetServerGroups通过调用callHttp()获取codis在zookeeper上注册的server groups信息

PingServer用来检测codis server也就是redis服务的状态，并将CheckAlive的结果放进一个channel

verifyAndUpServer来确定一个offline的redis实例是存活的并将其设置为online状态

getSlave: 当一个redis master挂掉的时候，用来获取同一个group下的slave

handleCrashedServer用来处理挂掉的redis，如果挂的redis是master则获取该codis group对应点slave并将其promote到master，其他则不作处理

handleAddServer将存活的redis添加其对应给group下作为slave节点

CheckAliveAndPromote是codis-ha的核心方法，解析见代码注释

####最后是main.go
main.go除了程序入口外还定义了一些全局变量和一个包函数。

总的来说，codis-ha做了下面三件事：

- 获取codis server groups列表
- 检查codis server(redis实例)的状态
- 检查offline状态的codis server并将其上线(online)

Linux中毒（10字符文件名）

Posted on Dec 8 2015   |   In OS , Linux   |  

背景

对Linux本来只限于简单应用，由于现在创业公司没有运维，很多事情要自己上。昨天收到阿里云警告说我们的服务器在对外DDOS，限时解决。

上去一看tomcat地下多了个A.war的项目，基本确定因为使用tomcat自带的manager调试的时候被黑了。

情景

实例目录下有个mg24的程序在跑，以为杀掉就可以了，我想的真的是太简单了。警告依然没有接触，而且mg24这个玩意又产生了，看了一下用户排除root被拿到，于是先改密码。

检查/etc/crontab

发现cron.hourly下面有个/lib/libudev4.so的文件，按照网上说的file /lib/libudev4.so看到的确是个可执行程序，也就是病毒了。可是直接删也删不掉。init.d地下也出现了异常启动脚本，发现是/usr/bin低下的一个文件。

ps也没看到异常进程，查看一下ls -lh /bin/ps 发现ps命令被黑了，同时还有netstat。网上查一下，有类似的案例，叫10字符病毒程序，参见附录。跟我的情况基本一样，我按照他的操作处理了一下发现没用。杀掉就重启，病毒文件重新生成。十字符文件还换者地方产生，chattr +i /usr/bin， 它就跑到/bin下， chattr +i /bin就跑到/tmp下，好烦。因为我是运维小白，各种网上搜索终于解决了。下面总结下:

解决

  1. 还原 ps netstat等所有被黑的命令， 网上说有备份(cp /usr/bin/dpkgd/ /sbin/ && cp -f /usr/bin/dpkgd/ /bin/)

  2. 查看病毒位置，进程id等，还原rc.d下的文件，还有crontab，去掉恶意文件执行权限

  3. 不要直接杀掉进程，否则会再产生(比如pid是1234)

  4. 清除/etc/rc.d下所以病毒文件，可能在之前杀掉进程的时候产生了很多新的病毒文件，记住都要删掉

  5. 这时候再杀掉病毒进程，删除病毒本体。

总结
